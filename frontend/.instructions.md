# Frontend — Copilot Instructions

You are working in the **frontend** of Zault, a React + TypeScript application built with Vite.

## Key Context

- Read `CONTEXT.md` (root) for system architecture.
- Read `frontend/CONTEXT.md` for frontend-specific decisions and state.
- **Read `docs/api/openapi.yaml` for all API endpoints, request/response shapes.**
- Generated TypeScript types are in `frontend/src/api/schema.d.ts`.
- For full design reference (color tokens, D3 charts, animation, spacing) → read `frontend/design.md`.

## Do NOT

- **Do NOT read backend Java code.** Use `docs/api/openapi.yaml` as the single source of truth for the API.
- Do NOT suggest yarn or pnpm — this project uses npm.
- Do NOT add `rounded-*` classes or border radius anywhere.
- Do NOT use decorative color — greyscale only. Color is reserved for semantic meaning (gain/loss/warning).
- Do NOT create ad-hoc button styles — use existing themed components.

## Visual Language (Mandatory)

This project has a **brutalist notebook** aesthetic. These rules are non-negotiable:

- **Bracket notation:** All button and link labels render as `[ Label ]` — the brackets are part of the visible text.
- **Zero border radius:** Never add `rounded-*` classes. All elements have sharp corners.
- **Invert on hover:** Interactive elements invert to `bg-foreground text-background` on hover.
- **Monospace only:** JetBrains Mono for all text. Never add sans-serif fonts.
- **No decorative color:** Palette is greyscale. Color appears only for gain (green), loss (red), warning (amber).
- **Underline inputs:** Inputs use bottom-border only — no bordered boxes, no background.
- **Sketch lines (light mode only):** Faint construction marks on containers via `.sketch-lines` class. Dark mode shows no lines or shadows.
- **Headings use `#` prefix:** Page titles render as `# Title`, sections as `## Section`.
- **Tooltips:** All tooltips use `bg-background text-foreground border border-foreground` — no border radius, no decorative backgrounds. Use the Radix `TooltipContent` component for interactive elements; use a custom positioned div with the same classes for charts/SVG.

## Components

Before creating new UI elements, check the **Component Inventory** in `frontend/design.md`.

**Key rule:** Use `BracketButton` for visible action buttons, `SubmitButton` for form submits. Only use raw `<Button>` for icon-only or ghost/asChild cases.

## Conventions

- Functional React components, TypeScript strict mode
- API base URL: `/api/` (proxied by nginx in production, Vite proxy in dev)
- CLI commands: use `make` targets (see root `Makefile`)
- Regenerate API types: `make api-types`

## API Types

Types are auto-generated from the OpenAPI spec:
```bash
make api-types  # generates frontend/src/api/schema.d.ts
```
Import and use these types instead of defining API shapes manually.

## Verification (After UI Changes)

Before finishing any UI work, verify:

- [ ] Button/link labels wrapped in `[ ]` brackets (via `BracketButton` or `SubmitButton`)?
- [ ] No `rounded-*` classes anywhere?
- [ ] Hover states invert colors (handled by Button component)?
- [ ] Reusing existing themed components — not raw HTML or ad-hoc styles?
- [ ] Color used only for gain/loss/warning semantics — no decorative color?
- [ ] Inputs use underline style (`FormField` or `Input` component)?

## Maintaining `frontend/CONTEXT.md`

`CONTEXT.md` is the living state document — it tracks what exists and why (not rules, which live here). Update it when:

- **Adding a new page or feature** → update "Current State"
- **Making a technology decision** (e.g., choosing a state management library) → update "UI Decisions"
- **Incurring known shortcuts** → add to "Technical Debt"

Do NOT duplicate rules from this file into CONTEXT.md. Keep CONTEXT.md factual and concise.
